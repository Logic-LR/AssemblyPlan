# 装配树生成：方法、结果与分析

最后更新：2026-05-27

---

## 1. 背景

### 任务

给定 N 个原始零件（几何 + 每个零件的 SVG 特征），预测装配树。

```
输入:  [part_0, part_1, ..., part_N]  （几何 + SVG 形状 + 空间特征）
输出:  装配树  例如 [[0, 1, 2], 3]  （先合并 {0,1,2}，再加入 3）
```

### 数据

- 102 个宜家家具对象（73 个训练，29 个测试）
- 754 个原始零件，404 个树动作
- 393 个手动步骤，每个步骤包含简化 SVG 实例
- 每个手动步骤：`parts`（活跃簇）+ `connections`（显式连接边）
- GT 装配树 + tree_actions_postorder

### 评估指标

- **Simple F1**：预测的非叶节点零件集合是否与 GT 匹配？
- **Hard F1**：预测的节点子节点（作为集合，忽略顺序）是否与 GT 匹配？

### 原始 IKEA-Manual 论文方法

```
方法:  DGCNN 提取每个零件的 1024 维点云特征
         → 对特征进行 K-Means（k=2,3）→ 轮廓系数选择最佳 k
         → 最佳簇 = 子装配体 → 对剩余零件递归
         → 自顶向下递归构建树

模型:   0 个学习参数（纯基于特征的聚类）
训练:   无（无监督）
输入:   每个零件的点云 → DGCNN → 1024 维特征
输出:   装配树（通过递归聚类）

关键洞察: 形状相似性隐式编码装配层次结构。
            四条桌腿看起来相似 → 先聚类在一起。
            不同形状的零件在更高层级聚类。

局限性: 完全不使用手动步骤的 SVG。
```

## 2. 我们尝试的所有方法

### 方法 1：BCE 上下文 MLP（最佳）

```
脚本:   train_tree_planner_context.py
模型:    MLP(938→192→192→1)，218K 参数
输入:    pair_feat = [rep(A), rep(B), |A-B|, A*B, rep(A∪B), global_ctx]，938 维
           global_ctx = [M, log(M), total_parts, mean/std/cluster_sizes] + mean/std(所有 cluster_reprs)
训练:    池化所有对象的所有步骤的所有配对 → 4500 个样本
          BCE 损失：正样本 = GT 树动作中的子节点-子节点配对
          输入特征的全局标准化
推理:    贪心：对所有配对打分 → 阈值 → 连通分量分组 → 合并
           阈值在验证集上调优（16 个值，最佳 Hard F1）
```

**结果：**

| 特征模式 | 测试 Simple | 测试 Hard | 全部 Hard |
|---|---|---|---|
| geometry | 0.412 | 0.109 | 0.335 |
| svg_geometry | 0.407 | 0.108 | 0.423 |
| svg_geometry_composite | **0.570** | **0.332** | **0.673** |

**5 种子划分方差（svg_geometry_composite）：**
```
s0=0.332  s1=0.303  s2=0.254  s3=0.136  s4=0.346
均值=0.274 ± 0.085
```

**问题：**

1. **66% 的扁平树预测**（29 个测试对象中的 19 个）。模型输出所有零件的单步合并。
2. **同一配对在不同步骤得到相同分数。** 在 applaro_2 上测试：
   - 配对 (0,2) 在步骤 0：分数 0.964
   - 配对 (0,2) 在步骤 1：分数 0.987 ← 几乎相同
   - 665 维局部配对特征在两个步骤中完全相同
   - 只有 273 维 global_ctx 不同，且差异极小（平均绝对差 = 0.40）
3. **特征冗余。** |A-B| 和 A*B 产生 266 维的成对乘积，大部分是噪声。
4. **标签粗糙。** GT 树动作中的所有子节点-子节点配对都被标记为 1，即使是在手动 SVG 中未直接连接的配对。
5. **Composite tokens 是 2 倍作弊。** svg_geometry_composite (0.332) vs svg_geometry (0.108)。Composite tokens 编码了手动子装配体标注 — 模型被直接告知了答案。
6. **推理时分数压缩。** 在测试时，所有配对概率聚集在 0.4-0.5。模型无法产生有区分度的分数。

**高分具有误导性的原因：**

```
BCE 分数 0.332 是因为：
  - 少数对象运气好（偶然正确合并）
  - 29 个对象中的 19 个得到扁平树 → Hard F1 = 0.00
  - 平均值被少数幸运对象拉高
```

### 方法 2：GRPO 与 SVG 衍生奖励

```
脚本:   train_tree_grpo.py
模型:    相同的 ContextMergeMLP（218K 参数）
          + 参考模型（冻结副本，用于 KL 惩罚）
训练:    组相对策略优化（GRPO）
          每个对象：采样 K=8-16 棵树，计算奖励，在组内归一化
          带裁剪重要性采样的策略梯度
          朝向参考模型的 KL 惩罚
奖励:    r = w1 * SVG_coherence + w2 * spatial_SVG + w3 * GT_F1
          SVG_coherence：出现在任何手动步骤中的非叶节点比例
          spatial_SVG：每步 SVG 几何（邻近性 + 轴对齐 + 连接候选）
推理:    与 BCE 相同的贪心 CC 解码器
```

**变体：**

| 变体 | 描述 | 结果 |
|---|---|---|
| 从 BCE 热启动 | BCE 模型 → GRPO 微调，β=0.05 | 测试 Hard 0.306 |
| 从头训练 K=8，τ=3.0 | 随机初始化，高温探索 | 测试 Hard 0.170 |
| 从头训练 K=16，τ=1.5 | 更多样本，中等温度 | 测试 Hard 0.258 |
| 从头训练 + 空间奖励 | 在奖励中加入空间 SVG 几何 | 测试 Hard 0.281 |
| 热启动 + 空间 | BCE 初始化 + 空间奖励 | 测试 Hard 0.254 |

**多种子划分运行（svg_geometry_composite）：**

| 实验 | 测试 Hard |
|---|---|
| BCE 100% 标签 | 0.274 ± 0.085 (n=5) |
| BCE 10% 标签 | 0.184 ± 0.044 (n=3) |
| BCE 25% 标签 | 0.218 ± 0.049 (n=3) |
| GRPO 10% 标签 | **0.237**（+29% 超过 BCE） |
| GRPO 25% 标签 | **0.274**（+5% 超过 BCE） |
| GRPO 仅 SVG（0% GT，纯 SVG 奖励） | ~0.346（3 种子均值） |

**问题：**

1. **热启动时的探索失败。** BCE 训练后的 logits 接近确定性（sigmoid ≈ 0 或 1）。
   - 在 τ=1.0 时，只有 15-20% 的采样树是唯一的
   - 在 τ=5.0 时，只有 44% 唯一；τ=8.0 → 52% 唯一
   - 熵正则化（λ=0.1）几乎无帮助（20% → 22%）
   - Top-K 采样使情况更糟（限制为模型的首选选择）

2. **从头训练时的过拟合。** 随机初始化 → 高熵 → 良好的探索。
   - 验证 Hard 达到 0.509（超过 BCE 的 0.442）
   - 测试 Hard 仅 0.281（巨大差距）
   - 模型找到了在 62 个训练对象上有效但无法泛化的策略

3. **奖励信号问题。**
   - SVG coherence：许多不同的树得到相同的奖励 → 无区分能力
   - Spatial SVG：GT 树得分 0.17-0.33，随机树 0.0 → 有效信号但较弱
   - Pearson r(SVG_coherence, GT_F1) = 1.0（小对象上），但采样的唯一树非常少

4. **训练-测试不一致。** GRPO 训练策略以最大化奖励，但推理使用贪心 argmax。
   奖励信号在测试时被丢弃。

5. **推理时缺少空间特征。** 模型训练时使用了每步 SVG 空间特征（center_delta、axis_alignment、proximity），但推理时传入空的 spatial_map={}。模型从未学会在没有这个"拐杖"的情况下运作。

6. **每个对象的优化器步骤导致不稳定。** 损失剧烈波动（1.2 → 141.5），因为 optimizer.step() 是每个对象调用一次，而不是每个 minibatch 调用一次。

### 方法 3：Transformer 集合上下文树解码器

```
脚本:   train_tree_decoder.py
模型:    SetContextTreeDecoder
          input_proj: Linear→LayerNorm→ReLU
          编码器: 2 层 TransformerEncoder（128 维，4 头）
          pair_scorer: Linear(512→128→64→1)
训练:    通过 GT 树动作进行教师强制
          每一步：用 transformer 编码所有当前簇
          对所有配对打分 → BCE 损失（正样本 = 子节点配对）
推理:    相同的贪心 CC 解码器
```

**结果：在 73 对象数据集上表现不如扁平 MLP。**
（具体数字未记录 — 模型很早就放弃了。）

**问题：**

1. **数据不足。** 73 个对象 × ~4 个动作 = ~300 个训练状态。
   Transformer 注意力需要数千个样本才能学习有意义的模式。
   只有 300 个状态，注意力权重基本上是随机的。

2. **任务的不必要复杂性。** 当数据集如此小时，在 4-12 个簇上的自注意力相对于扁平 MLP 提供的收益极小。

### 方法 4：步骤条件化序列规划器（v1）

```
脚本:   train_step_planner.py (v1)
模型:    StepMergePlanner
          PartEncoder: shape_emb(4→16) + spatial_proj(9→32→16) → 每个零件 32 维
          ClusterEncoder: MultiheadAttention(32, 4 头) → Linear(32→hidden_dim)
          GRU(256→256): 跟踪步骤进度
          step_emb(20→32): 显式步骤计数器
          pair_scorer: Linear(773→256→128→1)
          总计: 640K 参数
训练:    通过 GT tree_actions_postorder 进行教师强制（~250 个状态）
          每一步：编码簇 + GRU 状态 + step_idx + 可选空间特征
          配对的 BCE 损失（正样本 = 当前动作的子节点）
          随机 50% 空间特征 dropout（强制 GRU 学习状态）
推理:    GRU 跟踪状态 → 配对打分 → 贪心 CC 合并
```

**结果：**
- 最佳验证 Hard：0.127（第 40 轮，然后退化）
- 测试 Hard：0.022
- 加入 step_emb 后：分数标准差从 0.07 提升到 0.21，但仍然是扁平树

**问题：**

1. **只有 250 个训练状态。** 62 个对象 × ~4 个树动作 = ~250 个状态。
   每个状态在每个对象中只出现一次。GRU 无法从 250 个样本中学习有意义的状态转换。

2. **推理时没有空间特征。** 训练时使用了每步 SVG 空间特征（center_delta, distance, axis_alignment）。在推理时，spatial_map={}。
   模型学会了依赖空间特征作为"拐杖" — 没有它们，分数崩溃。

3. **50% 空间 dropout 不足。** 即使有 dropout，GRU 也无法学会仅从簇特征编码"我在哪一步？"。

4. **活跃性头没有帮助。** 加入了对"哪些簇在此步骤中活跃？"的预测。
   训练标签：当前树动作的子节点。但额外的损失项没有改善区分度。

### 方法 5：步骤条件化序列规划器（v2 — 连接）

```
脚本:   train_step_planner.py (v2，重写)
模型:    StepPlanner（比 v1 更简单）
          shape_emb(4→16) + spatial_proj(8→32→16)
          cluster_attn → GRU(256) + step_emb(20→32)
          pair_scorer: Linear(800→256→128→1)
训练:    393 个手动步骤，每个步骤的 CONNECTIONS 作为正配对标签
          通过手动步骤序列进行教师强制
          配对的 BCE 损失
推理:    GRU + step_emb → 配对打分 → CC 合并
```

**结果（5 个 epoch）：**
- 测试 Hard：0.021（仍然是扁平树）
- 损失：0.254 → 0.178（递减）

**问题：**

1. **连接不编码层级结构。** 所有手动步骤的连接在模型看来结构相似 — 带边的簇配对。
   模型学会了"这些是连接的"，但没有学会"这些在步骤 0 连接，这些在步骤 3 连接。"

2. **62 个对象上仍然只有 393 个状态。** 对于 GRU 泛化来说，多样性不足。

3. **GRU 输入过于压缩。** GRU 接收 mean(cluster_embeddings) — 一个单一向量。
   它无法仅从这个均值区分"6 个单例"和"2 个大簇"。

### 方法 6：聚类 + 对比学习

```
脚本:   train_cluster_planner.py
模型:    PartProjector: Linear(8→128→128→64)，50K 参数
训练:    连接的对比损失
          正样本：在嵌入空间中拉近已连接零件配对
          负样本：推远未连接零件配对（> margin）
          Hinge 损失：pos_dist.mean() + clamp(margin - neg_dist, 0).mean()
推理:    K-Means(k=2,3) + 轮廓系数 → 递归自顶向下树
          （与原始 IKEA-Manual 论文相同的算法）
```

**结果（30 个 epoch）：**
- 测试 Hard：0.035（与仅几何贪心基线相同）
- 对比比率：已连接配对比未连接配对近 2.5 倍
- 损失递减：12.46 → 0.60

**问题：**

1. **对比损失使层级结构扁平化。** 所有步骤的所有连接被平等对待。
   零件 {0,2,3,4} 在步骤 0 连接，然后在步骤 1 与 {5} 连接，然后在步骤 2 与 {1} 连接。
   所有 6 个零件最终都靠得很近 → K-Means 无法找到有意义的分割。

2. **形状相似性与连通性。** 原始论文使用编码形状相似性的 DGCNN 特征（四条桌腿看起来相似 → 聚类在一起）。
   我们的对比嵌入编码了连通性（所有连接的零件都靠得近）。
   这些根本不同 — 连通性不编码装配层次结构。

3. **K-Means 是无参数的。** 无法将步骤顺序或手动知识注入聚类过程。轮廓系数是装配树质量的糟糕代理。

## 3. 总结：根本原因

### 为什么我们的所有方法都表现不佳

| 根本原因 | 受影响的方法 | 解释 |
|---|---|---|
| **未使用步骤顺序** | 方法 1-6 | 所有方法都在步骤间池化或平均。模型从未学会"先合并 {0,1,2}，然后 {0,1,2,3}"。 |
| **训练数据太小** | 方法 3-5 | 62 个对象 × ~4 个树动作 = ~250 个状态。GRU/Transformer 需要数千个状态才能学习有意义的转换。 |
| **特征编码"什么"而非"何时"** | 方法 1-6 | 零件特征告诉模型哪些零件相似，但不告诉它们应该在哪一步合并。 |
| **推理-训练不匹配** | 方法 2, 4, 5 | 训练使用步骤 SVG（空间上下文），推理不使用。模型学会了错误的依赖关系。 |
| **扁平交叉熵无法编码层级结构** | 方法 1, 4, 5 | 每个配对的 BCE 平等对待所有步骤。应该在步骤 1 合并的配对与应该在步骤 3 合并的配对看起来相同。 |
| **标签粒度** | 方法 1 | GT 树动作中的所有子节点-子节点配对都是正样本，即使它们在手动 SVG 中未直接连接。 |
| **贪心 CC 解码器过于简单** | 方法 1-5 | 单一阈值控制一切。如果分数被压缩，CC 要么产生扁平树（低阈值），要么什么都不产生（高阈值）。没有中间地带。 |

### 我们拥有的有效方法

1. **Composite tokens 非常强大。** 它们直接编码手动子装配体知识。测试 Hard 0.332 vs 没有它们时的 0.162。但它们是"作弊" — 手册告诉了你哪些零件形成子装配体。

2. **当标签稀缺时，GRPO 有帮助。** 在 10% GT 标签时，GRPO + SVG 奖励比单独使用 BCE 高出 +30%。这是一个真正的贡献 — SVG 可以替代缺失的 GT 树标签。

3. **SVG 空间奖励是有效信号。** GT 树得分 0.17-0.33，随机树 0.0。信号存在 — 我们只是还没有弄清楚如何有效地优化它。

4. **步骤条件化模型消除了扁平树。** 序列 GRU 模型产生 0% 的扁平树率（相比之下 BCE 为 66%）。它们产生了错误的树，但至少产生了树。

### 原始论文做对的地方

1. **形状相似性自然编码层级结构。** 相似零件在较低层级聚类。
2. **递归聚类原生构建树。** 不需要序列预测。
3. **无学习参数意味着无过拟合。** 适用于任意数量的对象。
4. **K-Means + 轮廓系数简单且鲁棒。** 无阈值调优，无梯度问题。

## 5. 开放问题

1. **我们能否在不使用序列预测的情况下利用手动步骤顺序？** 步骤嵌入 + BCE 训练保留了 BCE 的泛化能力，同时增加了步骤感知。尚未测试。

2. **我们能否获得更好的零件特征？** 原始论文使用 DGCNN（1024 维点云特征）。我们使用手工制作的 36 维特征。能否用 part_images/（754 个渲染的 OBJ 视图）训练更好的特征提取器？

3. **贪心 CC 解码器是瓶颈吗？** 我们所有模型都使用相同的解码器。即使使用当前特征，学习过的解码器（RNN/指针网络）会有帮助吗？

4. **数据增强有多少帮助？** 102 个对象很小。我们能否从 393 个手动步骤生成合成装配序列？
